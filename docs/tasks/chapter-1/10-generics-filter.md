# Задача 10. Дженерики: `Filter[T]`

## Связанные темы
- 1.12.1 Введение в дженерики: параметры типа и ограничения (constraints)
- 1.12.2 Пакет constraints и создание собственных ограничений (union, ~T)
- 1.12.3 Рекурсивные ограничения типов
- 1.12.4 Практические паттерны применения дженериков (и когда они избыточны)

## Цель
Реализовать обобщённую утилиту `Filter[T]` для работы со срезами проекта (фильтрация истории переходов, фильтрация ссылок по TTL и т.п.). Продемонстрировать параметры типа, ограничения, `~T` и случай, когда дженерики избыточны.

## Что нужно сделать

### 1. Функция `Filter[T]` (`internal/shortener/filter.go`)
```go
// Filter возвращает новый срез, содержащий только те элементы items,
// для которых pred(item) == true. Исходный срез не модифицируется.
func Filter[T any](items []T, pred func(T) bool) []T {
    out := make([]T, 0, len(items))
    for _, v := range items {
        if pred(v) {
            out = append(out, v)
        }
    }
    return out
}
```
- Параметр типа `T` с ограничением `any` (тема 1.12.1).
- `pred` — функция-предикат от `T`.

### 2. Сопутствующие обобщённые функции
- `Map[T, U any](items []T, fn func(T) U) []U` — преобразование каждого элемента.
- `Reduce[T, U any](items []T, init U, fn func(U, T) U) U` — свёртка.
- Использовать их в демонстрации: например, `Map` истории переходов в срез кодов.

### 3. Собственное ограничение с `~T` (тема 1.12.2)
- Завести ограничение для числовых типов:
  ```go
  type Number interface {
      ~int | ~int64 | ~float64
  }
  // Sum возвращает сумму числовых элементов.
  func Sum[T Number](items []T) T
  ```
- Демонстрация: тип `type Counter int64`, удовлетворяющий `~int64`, передаётся в `Sum` (без `~` это не сработало бы).

### 4. Рекурсивные ограничения (тема 1.12.3)
- Демонстрация ограничения, ссылающегося на себя (например, `comparable`-подобный или строковое представление):
  ```go
  type Stringer interface {
      String() string
  }
  // JoinStrings объединяет элементы, реализующих Stringer.
  func JoinStrings[T Stringer](items []T) string
  ```
- Комментарий: рекурсивные ограничения (через `~` и интерфейсы, ссылающиеся на себя) — продвинутая тема; в проекте достаточно базовых.

### 5. Применение в проекте
- `HistoryByCode(code string) []Visit` (из задачи 03) переписать через `Filter`:
  ```go
  func (m *MemoryStorage) HistoryByCode(code string) []Visit {
      return shortener.Filter(m.history, func(v Visit) bool { return v.Code == code })
  }
  ```
  (Или вынести `Filter` в общий пакет `internal/xslices`, чтобы избежать цикла импортов.)
- Фильтрация активных (не истёкших) ссылок через `Filter[*Link]` + `IsExpired`.

### 6. Когда дженерики избыточны (тема 1.12.4)
- Комментарий в `filter.go`: дженерики уместны, когда алгоритм не зависит от типа (`Filter`, `Map`, `Reduce`, `Sum`). Если есть одна конкретная операция над одним типом — обычная функция проще и читаемее. Не вводите дженерики «для красоты».

## Требования к коду
- Параметры типа с осмысленными именами (`T`, `U`) или содержательными (`Number`, `Stringer`).
- Ограничения объявлены через `interface{}` с union/`~T` где уместно.
- Комментарии на русском объясняют, когда дженерики оправданы.

## Подсказки / подводные камни
- `any` — алиас `interface{}`; используйте как ограничение по умолчанию.
- `~T` разрешает типы с тем же underlying-типом (например, `type Counter int64`); без `~` — только сам `int64`.
- Не плодите дженерики там, где достаточно `[]*Link` и обычной функции.
- Инстанциация `Filter[Visit](...)` явная, но часто выводится автоматически — `Filter(items, pred)`.
- `comparable` — встроенное ограничение для типов, поддерживающих `==`/`!=` (нужно для ключей map).
- Цикл импортов: если `Filter` в `internal/shortener`, а `MemoryStorage` в `internal/store` его использует, `store` будет зависеть от `shortener` — нарушение направления. Вынесите `Filter` в нейтральный пакет (`internal/xslices` или `internal/util`).
